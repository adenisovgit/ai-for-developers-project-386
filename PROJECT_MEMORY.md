# Состояние проекта (Project Memory)

Документ для фиксации текущего состояния проекта, архитектурных решений и рабочих договоренностей.

## Общее состояние
- Инициализирован монорепозиторий.
- Правила для AI находятся в `AGENTS.md`.
- В проекте уже есть `api/` и `frontend/`.
- Во frontend реализованы публичный сценарий бронирования и панель владельца.

## API
- Источник правды для контракта: `api/main.tsp`.
- `api/openapi.yaml` должен генерироваться из TypeSpec и не редактируется вручную.
- `GET /event-types` возвращает список типов событий.
- `POST /event-types` использует отдельную модель `EventTypeCreateRequest`.
- `GET /slots` принимает только `startDate` и `endDate`.
- `GET /slots` не принимает `eventTypeId`.
- `GET /slots` возвращает только занятые интервалы `startTime` / `endTime`.
- `POST /bookings` принимает `eventTypeId`, `startTime`, `guestName`, `guestEmail`, `comment?`.
- `GET /bookings` возвращает список бронирований.

## Доменные правила
- `availableFrom` и `availableTo` задаются только значениями `HH:00`.
- Это ограничение зафиксировано в домене, TypeSpec и frontend-валидации.
- Тип события нужен для:
  - отображения информации о звонке
  - определения `durationMinutes`
  - определения окна доступности `availableFrom` / `availableTo`
  - передачи `eventTypeId` при создании бронирования
- Занятость для шага выбора времени запрашивается только после явного выбора даты.
- Пустой ответ `/slots` означает, что на выбранный день нет занятых интервалов.

## Построение слотов
- Дневная таблица слотов строится на фронтенде.
- Базой служат:
  - `durationMinutes`
  - `availableFrom`
  - `availableTo`
  - массив занятых интервалов из `/slots`
- Сетка фиксирована и не сдвигается от конца занятого интервала.
- Стартовые точки кратны `durationMinutes`.
- Слот считается занятым при любом пересечении с busy interval.
- В публичном интерфейсе показываются только доступные слоты.

## Frontend
- Frontend реализован на Vue 3 + TypeScript.
- HTTP-клиент и модели лежат в `frontend/src/api`.
- `frontend/src/api` генерируется из `api/openapi.yaml`.
- Сгенерированные файлы во frontend нельзя править вручную.
- Публичный flow:
  - шаг 1: выбор типа события
  - шаг 2: выбор даты и доступного слота
  - шаг 3: ввод контактных данных
- После выбора слота фокус автоматически переходит в поле `Имя`.

## Mock и Tooling
- Для mock API используется Prism поверх `api/openapi.yaml`.
- Prism должен запускаться без `--dynamic`, чтобы использовать examples из OpenAPI.
- Рабочая цепочка генерации такая:
  - обновить `api/main.tsp`
  - пересобрать `api/openapi.yaml`
  - перегенерировать `frontend/src/api`
- Нейтральные workflow-документы лежат в `docs/agent-playbooks/`.
- Для совместимости могут существовать симлинки из `.codex/skills/` на эти документы.

## UX договоренности
- Вверху приложения есть единая панель высотой 40px.
- Слева встроенный сегмент `Бронирование / Панель владельца`.
- Справа встроенный сегмент выбора темы.
- Активные сегменты в шапке не должны иметь hover-эффектов с подскоком или смещением.
- В выборе времени слоты показываются как компактные варианты по flex, без отдельной подписи `Свободно`.

## Процесс работы
- Перед внесением существенных изменений нужно быстро сверяться с этим документом.
- При изменении архитектуры, контракта, инфраструктуры или бизнес-правил этот документ нужно обновлять.
