# Call Booking Frontend

Используйте этот playbook при изменениях booking UI и owner UI во frontend.

## Public booking flow

1. Шаг 1: выбор типа события.
2. Шаг 2: выбор даты и доступного слота.
3. Шаг 3: ввод контактных данных.

## Важное поведение

- Выбор типа события не должен запускать загрузку слотов.
- Загрузка слотов начинается только после явного выбора даты.
- После выбора слота фокус должен переходить в поле `Имя`.

## Поведение слотов

- API возвращает только busy intervals на выбранный день.
- Frontend строит дневную сетку сам.
- Входные данные для построения сетки:
  - `durationMinutes`
  - `availableFrom`
  - `availableTo`
  - busy intervals из `/slots`
- Стартовые точки фиксированы и привязаны к окну доступности.
- Слот недоступен при любом пересечении с busy interval.
- В публичном UI показываются только доступные слоты.

## Mock-режим

- Prism должен использовать examples из OpenAPI, а не dynamic random values.
- Пустой ответ `/slots` означает, что на выбранный день нет занятых интервалов.

## UX-договоренности

- Вверху приложения есть компактная панель высотой 40px.
- Слева расположен встроенный переключатель маршрутов.
- Справа расположен встроенный переключатель темы.
- Активные сегменты в шапке не должны подпрыгивать, подниматься или смещаться.
- Выбор слотов отображается как компактный flex-набор, а не как таблица статусов.
