# Доменная модель и сценарии проекта "Запись на звонок"

### 1. Доменные Сущности

**Тип события (EventType)**
Шаблон для встречи, который создает владелец. Гость выбирает один из таких типов, чтобы забронировать звонок.
- `id`: UUID
- `title`: Строка (Название, например, "Знакомство")
- `description`: Строка (Описание события)
- `durationMinutes`: Число (Длительность, например: 15, 30, 45, 60)
- `availableFrom`: Время (Начало окна доступности, например, "08:00")
- `availableTo`: Время (Конец окна доступности, например, "16:00")

**Слот (Slot)**
Свободное окно в календаре владельца, доступное для бронирования гостем для конкретного типа события. Рассчитывается динамически.
- `startTime`: Дата/Время в формате UTC

**Бронирование / Встреча (Booking)**
Фактическая запись гостя на конкретное время по определенному типу события.
- `id`: UUID
- `eventTypeId`: Ссылка на выбранный тип события
- `startTime`: Дата/Время в формате UTC
- `guestName`: Имя гостя
- `guestEmail`: Почта гостя
- `comment`: Обязательный комментарий от гостя

**(Неявная сущность) Владелец**
Поскольку регистрации и авторизации на уровне бэкенда нет, профиль владельца — один на всю систему. Отдельной сущности и эндпоинтов для CRUD владельца не требуется.
> [!NOTE]
> Разделение прав доступа между владельцем и гостями производится исключительно на уровне роутинга фронтенд-приложения (разные URL-маршруты/страницы), серверная часть повністю публична.

### 2. Поддерживаемые Сценарии и Передаваемые Данные

**Сценарий 1: Просмотр видов брони (Гость)**
- **Действие:** Гость заходит на страницу со списком вариантов звонка.
- **Эндпоинт:** `GET /event-types`
- **Передается:** Никаких параметров.
- **Возвращается:** Массив типов событий (`id`, `title`, `description`, `durationMinutes`, `availableFrom`, `availableTo`).

**Сценарий 2: Поиск доступного времени (Гость)**
- **Действие:** Гость выбрал тип события и хочет посмотреть свободные слоты.
- **Эндпоинт:** `GET /slots`
- **Передается (Query):**
    - `eventTypeId`
    - `startDate`, `endDate`
- **Возвращается:** Массив объектов `Slot` (содержащих `startTime`).

**Сценарий 3: Создание бронирования (Гость)**
- **Действие:** Гость выбрал слот, заполнил форму и нажал "Забронировать".
- **Эндпоинт:** `POST /bookings`
- **Передается (Body):** `eventTypeId`, `startTime`, `guestName`, `guestEmail`, `comment`.
- **Возвращается:** Объект созданного `Booking`.

**Сценарий 4: Создание типа события (Владелец)**
- **Действие:** Владелец добавляет новый тип звонка.
- **Эндпоинт:** `POST /event-types`
- **Передается (Body):** `title`, `description`, `durationMinutes`, `availableFrom`, `availableTo`.
- **Возвращается:** Объект созданного `EventType`.

**Сценарий 5: Просмотр предстоящих встреч (Владелец)**
- **Действие:** Владелец смотрит все свои запланированные созвоны.
- **Эндпоинт:** `GET /bookings`
- **Передается:** Без параметров.
- **Возвращается:** Массив всех объектов `Booking`.

### 3. Ограничения

1. **Глобальное правило занятости (Единый Календарь):** Два бронирования ни при каких условиях не могут пересекаться по времени. Если уже есть занятый слот, ни один тип события не может быть забронирован так, чтобы его временной промежуток наложился на существующую встречу.
2. **Ограничение типа события (Локальное правило):** Свободные слоты для выбранного типа события генерируются строго в границах между его `availableFrom` и `availableTo`.
3. **Обязательные поля формы:** Гость обязан заполнить Имя, Email и Комментарий.
4. **Свободный доступ:** Без авторизации. Все операции полностью публичны.
