# Доменная модель и сценарии проекта "Запись на звонок"

### 1. Доменные сущности

**Тип события (EventType)**
Шаблон встречи, который создает владелец. Гость выбирает один из таких типов, чтобы забронировать звонок.
- `id`: UUID
- `title`: строка
- `description`: строка
- `durationMinutes`: одно из значений `15 | 30 | 45 | 60`
- `availableFrom`: время в формате `HH:mm`, только значение, кратное часу
- `availableTo`: время в формате `HH:mm`, только значение, кратное часу
- `color`: обязательная HEX-строка вида `#RRGGBB`

**Запрос на создание типа события (EventTypeCreateRequest)**
Отдельная модель для `POST /event-types`.
- включает все поля `EventType`, кроме `id`

**Занятый слот (OccupiedSlot)**
Занятый интервал времени, который приходит с API для выбранного дня.
- `startTime`: начало занятого интервала в UTC
- `endTime`: конец занятого интервала в UTC

**Бронирование (Booking)**
Фактическая запись гостя на конкретное время по определенному типу события.
- `id`: UUID
- `eventTypeId`: ссылка на выбранный тип события
- `startTime`: дата и время в UTC
- `guestName`: имя гостя
- `guestEmail`: почта гостя
- `comment`: необязательный комментарий гостя

**Запрос на создание бронирования (BookingRequest)**
- `eventTypeId`
- `startTime`
- `guestName`
- `guestEmail`
- `comment?`

**Владелец**
Регистрации и авторизации нет. В системе предполагается один владелец, а разделение сценариев выполняется только на уровне фронтенд-маршрутов.

### 2. Поддерживаемые сценарии и данные

**Сценарий 1: Просмотр типов событий (Гость)**
- **Эндпоинт:** `GET /event-types`
- **Передается:** без параметров
- **Возвращается:** массив `EventType`

**Сценарий 2: Поиск доступного времени (Гость)**
- **Эндпоинт:** `GET /slots`
- **Передается (query):**
  - `startDate`
  - `endDate`
- **Возвращается:** массив `OccupiedSlot`

Правило для этого сценария:
- на шаге 1 фронтенд получает только список типов событий и использует выбранный тип для длительности, окна доступности и последующего `POST /bookings`
- запрос занятости выполняется только после явного выбора даты на шаге 2
- для публичной страницы фронтенд передает один выбранный день
- API возвращает только занятые интервалы на выбранный день
- фронтенд строит полную дневную сетку самостоятельно
- шаг сетки равен `durationMinutes` выбранного типа события
- стартовые точки сетки всегда кратны `durationMinutes` и остаются привязанными к целому часу внутри окна доступности
- свободный или занятый статус каждой точки вычисляется на основании пересечения с занятыми интервалами
- если `availableFrom = 09:00`, `availableTo = 18:00` и `durationMinutes = 15`, то допустимые старты это `09:00`, `09:15`, `09:30`, `09:45`, `10:00` и далее
- если пришли занятые интервалы `09:01-09:31` и `11:14-12:28`, то слоты `09:00-09:15`, `09:15-09:30`, `09:30-09:45` считаются занятыми, а первый свободный слот будет `09:45-10:00`
- сетка не сдвигается от конца занятых интервалов и продолжает строиться по фиксированным шагам
- если backend учитывает буферы или перерывы, они должны быть уже заложены в занятые интервалы ответа

**Сценарий 3: Создание бронирования (Гость)**
- **Эндпоинт:** `POST /bookings`
- **Передается (body):** `BookingRequest`
- **Возвращается:** созданный `Booking`

**Сценарий 4: Создание типа события (Владелец)**
- **Эндпоинт:** `POST /event-types`
- **Передается (body):** `EventTypeCreateRequest`
- **Возвращается:** созданный `EventType`

Дополнительные правила:
- владелец выбирает `availableFrom` и `availableTo` только из значений, кратных часу
- значения по умолчанию для окна доступности: `09:00` и `18:00`

**Сценарий 5: Просмотр предстоящих бронирований (Владелец)**
- **Эндпоинт:** `GET /bookings`
- **Передается:** без параметров
- **Возвращается:** массив `Booking`

---

### 3. Ограничения

1. **Глобальное правило занятости:** два бронирования не могут пересекаться по времени.
2. **Локальное правило типа события:** фронтенд строит сетку возможных стартов только в пределах `availableFrom` и `availableTo`.
3. **Выбор времени гостем:** гость выбирает время только из сетки слотов, без отдельного ручного ввода времени.
4. **Форма гостя:** обязательны `guestName` и `guestEmail`, `comment` остается необязательным.
5. **Свободный доступ:** все операции публичны, без авторизации.
