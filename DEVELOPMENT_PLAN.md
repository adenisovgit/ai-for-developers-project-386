# План разработки проекта «Запись на звонок» (Call Booking)

Этот файл содержит общий план проекта и используется как актуальный ориентир для следующих этапов разработки.

- **Шаг 4**: [x] Backend (Spring Boot)

## Шаг 1. Инициализация монорепозитория и подготовка AI-артефактов [ЗАВЕРШЕН]
**Цель:** Подготовить проект к работе с AI-агентами и настроить базовую структуру репозитория.

**Результаты:**
- Создана базовая структура монорепозитория.
- Добавлены project-level инструкции для агентов.
- Подготовлены директории `api/`, `frontend/` и `backend/`.

## Шаг 2. Проектирование API (TypeSpec) [ЗАВЕРШЕН]
**Цель:** Зафиксировать API-first контракт как источник правды для frontend и backend.

**Результаты:**
- Источник правды для API находится в `api/main.tsp`.
- Из TypeSpec генерируется `api/openapi.yaml`.
- Контракт покрывает сценарии типов событий, занятости слотов и бронирований.
- Контракт используется как база для генерации frontend API-клиента и backend DTO/интерфейсов.

## Шаг 3. Frontend (Vue 3 + Prism) [ЗАВЕРШЕН]
**Цель:** Реализовать пользовательский интерфейс публичного бронирования и панели владельца.

**Стек:** Vue 3, Vite, TypeScript, TanStack Query, VeeValidate, Prism.

**Результаты:**
- Во `frontend/` реализован публичный flow бронирования:
  - выбор типа события
  - выбор даты
  - выбор доступного слота
  - ввод контактных данных
- Реализована страница владельца `/admin` с просмотром типов событий и бронирований.
- Frontend использует сгенерированный API-клиент из `api/openapi.yaml`.
- Для frontend-разработки и контрактной проверки используется Prism mock server.
- UI уже поддерживает сценарий конфликта слота после ответа `409`.

## Шаг 4. Backend (Spring Boot) [ЗАВЕРШЕН]
**Цель:** Реализовать серверную часть, которая обслуживает реальный сценарий бронирования.

**Стек:** Java, Spring Boot, Gradle, in-memory хранилище.

**Результаты:**
- Реализован backend в `backend/`.
- Серверные DTO и интерфейсы генерируются из `api/openapi.yaml`.
- Работают эндпоинты типов событий, занятости слотов и бронирований.
- Добавлены доменные проверки:
  - валидность `eventTypeId`
  - принадлежность старта окну доступности
  - кратность стартового времени сетке `durationMinutes`
  - запрет пересечений бронирований
- Ошибка конфликта слота возвращается как `409`, а бизнес-ошибки маппятся в понятные plain-text ответы.

## Шаг 5. Интеграционное тестирование (Playwright)
**Цель:** Добавить автоматические интеграционные проверки, которые подтверждают, что ключевой путь бронирования проходит от браузера до backend и обратно.

**Стек:** `@playwright/test`, Chromium, GitHub Actions, Playwright MCP, Chrome Devtools MCP.

**Что нужно сделать:**
1. Зафиксировать основные пользовательские сценарии для e2e-проверки в общем плане и рядом с тестами.
2. Подключить Playwright во `frontend/` как тестовый раннер для браузерных сценариев.
3. Добавить `frontend/playwright.config.ts` с:
   - `baseURL`
   - таймаутами
   - `retries` для CI
   - `trace` при падении
   - `screenshot` при падении
   - `video` по необходимости для CI-диагностики
4. Подготовить тестовое окружение, в котором frontend работает против реального backend, а не против Prism.
5. Добавить npm-скрипты для локального запуска e2e в `frontend/package.json`.
6. Добавить root-level команду, например `make e2e`, которая поднимает backend и frontend в тестовом режиме и запускает Playwright.
7. Использовать **Playwright MCP** как обязательный инструмент подготовки и стабилизации тестов:
   - прогонять реальный пользовательский сценарий до написания финальных спеков
   - подбирать устойчивые локаторы
   - проверять переход фокуса в поле `Имя`
   - проверять успешное состояние после бронирования
   - воспроизводить конфликт слота
8. Использовать **Chrome Devtools MCP** вместе с Playwright MCP для диагностики:
   - ошибок в консоли
   - сетевых запросов к `/event-types`, `/slots`, `/bookings`
   - корректной работы повторной загрузки занятости после `409`
   - причин flaky-поведения, если такие появятся
9. После ручной проверки через MCP перенести подтвержденные сценарии в стабильные Playwright-спеки.
10. Настроить отдельный GitHub Actions workflow для интеграционных проверок.

**Обязательные сценарии:**
- Основной happy path бронирования:
  - открыть публичную страницу
  - выбрать тип события
  - выбрать дату
  - дождаться показа доступных слотов
  - выбрать слот
  - проверить, что фокус перешел в поле `Имя`
  - заполнить `Имя` и `Email`
  - отправить форму
  - увидеть успешное подтверждение бронирования
- Сквозная проверка результата:
  - перейти на `/admin`
  - убедиться, что новое бронирование отображается в списке владельца
- Сценарий конфликта слота:
  - сделать слот занятым параллельным действием
  - попытаться подтвердить устаревшее бронирование
  - убедиться, что frontend показывает конфликтный сценарий
  - убедиться, что frontend повторно запрашивает занятость и сбрасывает устаревший выбор слота
- Диагностические проверки через MCP:
  - на странице нет критических ошибок в консоли
  - frontend ходит в реальный backend, а не в Prism
  - после конфликта действительно уходит повторный запрос за занятостью

**Правила и границы шага:**
- `Playwright MCP` и `Chrome Devtools MCP` входят в рабочий процесс подготовки тестов, но не заменяют тестовый раннер.
- Источником истины для автоматического запуска остается `@playwright/test` и набор e2e-спеков в репозитории.
- Prism остается инструментом mock-разработки frontend, но не используется как среда для интеграционных e2e-тестов.
- Публичный API приложения на этом шаге не меняется.
- Для CI достаточно Chromium, без обязательной кроссбраузерной матрицы.
- Основной объем шага это end-to-end сценарий бронирования, а не полное покрытие всех owner-сценариев.

**Артефакты шага:**
- npm-скрипты e2e в `frontend/package.json`
- `frontend/playwright.config.ts`
- папка со спеками, например `frontend/tests/e2e/`
- отдельный workflow, например `.github/workflows/integration-e2e.yml`
- сохранение trace, screenshot и HTML report при падении тестов

**Ограничения по CI:**
- Существующий `.github/workflows/hexlet-check.yml` не удаляется и не редактируется.
- Новый workflow добавляется рядом и отвечает только за интеграционные Playwright-проверки.

**Результат шага:**
- В проекте есть автоматические Playwright-проверки, покрывающие основной путь бронирования.
- Тесты запускаются локально и в GitHub Actions.
- При падении доступны диагностические артефакты и понятный путь разбора через MCP-инструменты.

## Шаг 6. Контейнеризация (Docker)
**Цель:** Подготовить проект к развертыванию после того, как основной пользовательский путь будет закрыт интеграционными проверками.

**Задачи для выполнения в чате:**
1. Добавить `Dockerfile` для `backend/`.
2. Добавить `Dockerfile` для `frontend/`.
3. Подготовить `docker-compose.yml` для локального совместного запуска.
4. Убедиться, что контейнеризированный запуск совместим с выбранной стратегией интеграционного тестирования.
